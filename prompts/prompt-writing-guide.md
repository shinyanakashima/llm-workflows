# プロンプトの書き方ガイド

毎回同じ前提をコピペしなくて済むように、プロンプトを設計するための指針。

## 原則：不変はリポジトリ、可変だけ毎回書く

- 不変（方針・前提）は `preferences/` / `CLAUDE.md` / `AGENTS.md` / `skills/` に置き、参照させる。
- 可変（今回固有の情報）だけを毎回書く。
- 目標は、毎回タイプするプロンプトを 3〜5 行に収めること。

## 毎回書くプロンプトの型

可変部分だけを埋める骨格にする。

```txt
[役割] <どの Skill / 方針に従うか>
[目的] <何を、誰向けに、なぜ作る／検討するか>
[入力] <対象データ・文章・URL など>
[制約] <公開範囲・デプロイ先・期限など今回固有の条件>
[出力] <最初に何を返してほしいか／出力形式>
```

方針（MapLibre 優先・Vite 前提・出典明記など）は書かない。参照先にあるため。

## 良い例 / 悪い例

- 悪い: 方針を毎回すべて列挙する（長文化し、ブレやすい）。
- 良い: 「`preferences/` と `CLAUDE.md` を前提に、〇〇を作って。まず技術選定の確認から」。

## コピペしない仕組み

| ツール | やり方 |
| --- | --- |
| Claude Code / 各種エージェント | 各リポジトリ直下に `CLAUDE.md` / `AGENTS.md` を置く（自動で読まれる） |
| ChatGPT | `skills/` を `skill.zip` 化して登録する |
| 全般 | `prompts/` をスニペット展開ツールに登録し、短いトリガー語で呼び出す |

## 設計のコツ

1. 1 プロンプト = 1 目的（開発開始・レビュー・調査を混ぜない）。
2. 可変部分は穴埋め化する（`{{プロジェクト名}}` のような placeholder）。
3. 出力形式を固定する（毎回同じ構造で返るようにする）。
4. 参照先を明示する（「`preferences/tech-stack.md` に従って」の一言）。
5. 方針が変わったら `preferences/` だけ直す（プロンプトは参照なので追従する）。
